First, what does Airflow enable us to do? Allows you to build and run workflows. Every workflow is represented as a directed acyclic graph. Your workflow instances in Airflow are actually referred to as DAGs. Now, every task or bit of work that you want to perform is represented as a node or a [Vertex](../../Skills/Software%20Development/Vertex.md) in this directed acyclic graph. The edges that connect these nodes represent dependencies. Each directed edge is a dependency from one task to another. It also represents how data might flow between these dependencies. So what exactly is a directed acyclic graph? Here is an example. Observe this graph here where every edge has an arrow indicating direction. These directed edges typically represent the direction in which the data flows. These are actually used to represent dependencies in Airflow. The task that is pointed to depends on all of the previous tasks in the sequence, in the graph. Now, every node here represents some bit of work that needs to be performed, a task that needs to be executed. Now, these arrows also represent the dependencies. They need not necessarily represent the data flow but they typically do. That's why I mentioned that first. It's possible
>
> **[1:32](https://www.linkedin.com/learning/learning-apache-airflow/apache-airflow-concepts-and-architecture?u=76281980&t=92)** for a task in the graph to access data generated by any other previously executed task. Dependencies represent which task needs to be completed before another task can begin. For example, the email error task that you see here depends on the describe integrity task, and that in turn depends on all previous tasks that come before it in the graph. Now the graph cannot have cycles, because cycles represent circular dependencies, and in that case your workflow will never execute through to completion. This is why we needed directed acyclic graph. Here are the basic components that make up Airflow. Here is a big picture visualization. Now at the center of Airflow is the scheduler. The scheduler is what triggers the workflows and submits the workflow to the executors so that the individual tasks in the workflow can be run. The scheduler has to manage many important details before a task can be scheduled. It determines when a task should be executed, depending on what dependencies were completed, have all of the dependencies run through for this task. It'll also look at the availability of resources on your machine. It'll keep track of the tasks' state and handle task failures and any retries that you may have configured. The scheduler serves as the manager for Airflow. It's the orchestrator for Airflow's [Workflow Management](../../Skills/DevOps/Workflow%20Management.md) System. Next, we have the executor, whose job it is to receive tasks
>
> **[3:07](https://www.linkedin.com/learning/learning-apache-airflow/apache-airflow-concepts-and-architecture?u=76281980&t=187)** from the scheduler and then just execute those tasks. Now in the default installation, which is what we'll be using, the executor actually runs inside the scheduler. Now, this obviously won't work very well if you have a production workload with many complex heavy duty tasks. In production deployments the executor pushes the execution out to worker nodes. Airflow supports a number of different types of executors. The sequential executor is the default executor in Airflow, and it executes tasks sequentially in a single process. It's great for testing and debugging workflows. The local executor also executes tasks in a single process, but it can run multiple tasks in parallel on the same machine. This executor is the right one for small to medium-sized workflows that can be executed on a single machine. Then we have the Celery executor which uses the Celery distributed task queue to distribute task execution across multiple worker nodes. This is great for larger workflows. Airflow also supports a [Kubernetes](../../Skills/DevOps/Kubernetes.md) executor that uses the Kubernetes framework to create a pod for each task instance, which allows for efficient resource utilization and better [Scalability](../../Skills/Artificial%20Intelligence%20(AI)/Scalability.md). This is the right choice for containerized production workflows running on Kubernetes clusters. The web server component in Airflow runs a very useful user interface that we can use to inspect, trigger, and debug our DAGs and tasks. We'll be working with the web server throughout this course.
>
> **[4:43](https://www.linkedin.com/learning/learning-apache-airflow/apache-airflow-concepts-and-architecture?u=76281980&t=283)** Every workflow has some state, and it is this state that is stored in the [Metadata](../../Skills/Web%20Development/Metadata.md) database. This is used by the scheduler, executor, as well as the web server. All of your workflows will be implemented in [Python](../../Skills/Software%20Development/Python%20(Programming%20Language).md) and stored in the DAG directory. This is a folder of files which contain the DAG definitions and code, and this is the directory read by other components in Airflow. At this point, you have a good high level understanding of how Airflow works. I'm going to define two more terms. The first term is operators. Operators are predefined tasks that Airflow provides that can be used to build the nodes in your workflow directed Celery graph. We'll be working with Bash operators Python operators, SQLite operators for different use cases. And then finally, XComs. This stands for cross communication. This is a system that Airflow uses to pass data between tasks, where tasks can push and pull small bits of metadata. Values generated by by one task can be passed to other tasks in the workflow using XComs.

In Airflow, Catchup refers to the process of scheduling and executing all the past DAG runs that would have been scheduled if the DAG had been created and running at an earlier point in time. When Catchup is enabled, airflow will create DAG runs for all past intervals that have not yet been executed and will execute them in sequence until it reaches the current interval.
