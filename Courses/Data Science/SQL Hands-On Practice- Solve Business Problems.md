#### [Introduction to CTEs](https://www.linkedin.com/learning/sql-hands-on-practice-solve-business-problems/introduction-to-ctes?u=76281980)

> [!transcript]- Transcript
>
> **[0:00](https://www.linkedin.com/learning/sql-hands-on-practice-solve-business-problems/introduction-to-ctes?u=76281980&t=0)** - [Speaker] Common table expressions or [[CTE|CTEs]] for short are arguably one of the most powerful skills you can use in [[SQL]]. As I've grown in my career as a senior data analyst, most of the queries I write use at least one [[CTE]] so I definitely recommend you learn them as well. CTEs are a way to define a temporary result set that you can only use within the scope of that query. They are subsets of larger data sets and you can treat them just like a regular table. This means that you can perform functions on the data and even join them to other tables as usual. In other words, CTEs are like little data sets that you define first and then refer to later on in your main query. In the example we saw before, we created a CTE that aggregates the number of users by customer and then we found the average of all of the summed values. Since we can't nest aggregate functions in SQL such as taking the average of account, we must calculate the count in a CTE and then calculate the average in the main query below. The same logic also applies to other aggregate functions as well. Such as min, max and sum. Not only are we required to use a CTE here due to syntax requirements, it's also a lot easier to read and understand when we define the first result set first and then perform the final calculations later. If you're familiar with subqueries, CTEs are similar and most CTEs can also be written as subqueries and vice versa.
>
> **[1:34](https://www.linkedin.com/learning/sql-hands-on-practice-solve-business-problems/introduction-to-ctes?u=76281980&t=94)** If you're not familiar with subqueries, don't worry. You don't need to know them for this course but they may be useful to learn later on. CTEs do have some advantages over subqueries. The first advantage is that CTEs are reusable. This means that you can reference the CTE multiple times in the same query. For example, you can use a CTE inside of multiple other CTEs and in the main select statement at the end of the query. SQL remembers that CTE throughout your entire query. This is huge because it means that your work to create a CTE can be used multiple times instead of just once. The second advantage is that CTEs offer greater readability. Have you ever received code from a colleague that looked like a bowl of spaghetti soup and took hours of mental gymnastics to understand what the code is doing? Writing easily readable code is an essential skill to have for collaboration, code sharing and when advancing to more senior level roles. Since CTEs define other tables before the main query, it's generally a lot easier to understand the overall query especially when compared to subqueries, which are tangled and integrated into the main query. The third main advantage is that CTEs can be recursive which means that they can reference themselves. This is a skill used a lot in programming and it's one of the most advanced skills you can use in SQL. I personally haven't found a lot of use cases for this in my data analytics career,

> **[3:07](https://www.linkedin.com/learning/sql-hands-on-practice-solve-business-problems/introduction-to-ctes?u=76281980&t=187)** so it's not included in this course but it's still important to point out because subqueries don't have this capability. As you can see, CTEs have many advantages over subqueries and they can take your SQL game to soaring heights. So after the tie for customers B, C, and D, we have customer E. Customer E receives a rank value of five because there are four customers above it that ranked higher. So not only does rank consider the tie values when ranking the ties, it also considers the tied values for the values after the ties. The final window function used to rank values is dense rank. Dense rank also considers ties, and works almost the same way as rank. However, the only difference is that after a tie, dense rank will number the next rank value with the sequential number, leaving no gaps in the rank values. So in our example, customer E is the next customer after the tied values. Customer E receives a dense rank of three, because dense rank isn't going to skip any numbers. The differences between these three methods are fairly small, and I've mostly used row number on the job since I don't usually need to consider ties, but this is actually one of the most common SQL interview questions, so it's really important to know the differences between them and how to use them. Let's learn more about other window functions in the next video.

What do window functions do exactly? Window functions perform calculations across a specified set of rows or a window. Basically, the data is partitioned into pre-specified segments and the calculation is performed for each segment. They work similarly to regular aggregate functions, but instead of performing the aggregate across all rows, it's calculated multiple times, once for each partition of rows.
