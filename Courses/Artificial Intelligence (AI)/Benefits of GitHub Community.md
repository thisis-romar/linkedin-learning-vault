> **[0:00](https://www.linkedin.com/learning/benefits-of-github-community/what-is-open-source?u=76281980&t=0)** - Let's start with some definitions, and first is the concept of open source. Now this is code that is freely available for modification and collaboration online. With over 28 million open source repositories, [[GitHub]] is a great place to store your open source projects, and one of the reasons is [[Git]] itself. With other source control tools, the code for a project is in a centralized location, but with Git, everyone can have access to the entire code base running under a local machine or a machine in the cloud. That makes it super convenient to modify the code with others. Now to do this, GitHub gives you a formal process for submitting suggestions called pull requests, which helps you discuss suggestions for improvements with others. It also adds issues, which let people discuss potential changes or improvements within a community of developers. GitHub offers other collaboration tools, like discussions, projects, animation, and hosting for documentation. In order for a project to be truly open source, it should have a license which spells out how you can use and expand on the project. One of the keys to a popular open source project is to encourage sharing and collaboration, and GitHub has lots of ways to make your projects run smoother.

>
> **[0:00](https://www.linkedin.com/learning/benefits-of-github-community/what-is-innersource?u=76281980&t=0)** - Over time, open source has proven to be a successful method for creating and developing software. Some companies have adopted an approach internally that takes advantage of the main benefits of open source; we call this inner source. Companies often use and contribute to open-source projects internally so they're familiar with the added benefits of working with open source. Now, [[GitHub]] isn't just a place where open-source developers can collaborate. The application can run internally, providing some of the same features and values that are available within the universal version of GitHub. Part of creating an inner source culture means opening projects to a larger part of the organization. More collaboration means more opportunities for individuals within an organization to make contributions to your projects. Now, that opens up the ability for others to test the software and may help you find some new talent within the company which can become an important part of your development team. Another benefit of inner source is more transparency and visibility for your projects, allowing workers to have access to features as they're built and accepting comments and contributions from different departments. That approach ends up creating a more [[Agile Development|agile]] workflow that makes software better. Inner source doesn't mean that you'll sacrifice security. The version of GitHub you can use inside your organization respects things like firewall and security measures. So by adopting an inner source culture, you'll get the best of all worlds.

#### [What is Forking?](https://www.linkedin.com/learning/benefits-of-github-community/what-is-forking?u=76281980)

> [!transcript]- Transcript
>
> **[0:00](https://www.linkedin.com/learning/benefits-of-github-community/what-is-forking?u=76281980&t=0)** - One of the key strategies in an open source approach is the concept of forking. Forking allows a user to make a safe copy of a project while maintaining a connection with the original. Through forking, you share code and visibility settings, plus get credit to contributors adapting or modifying your code. Forks are traditionally used in a process called Gitflow where in order to make changes to a project, you start by forking it and then after making changes, create a pull request. The pull request allows discussions of the proposed changes and then upon acceptance, a way to merge the changes back into the main project. The forks also become a way of determining the popularity and activity of a project. So a project with a high number of forks denotes heavy activity within the project. The forks also maintain links to the original repositories and appear in the insights so that you can credit original works. By the way, forking is allowed in repositories by default, so if you want to allow people to make a copy of an existing repository without making them fork it, you can turn the repository into a template repository. Forking is one of the key strategies, which by providing a way for others to contribute, collaborate and discuss projects makes using [[GitHub]] a truly social experience.

#### [Discoverable repositories](https://www.linkedin.com/learning/benefits-of-github-community/discoverable-repositories?u=76281980)

> [!transcript]- Transcript
>
> **[0:00](https://www.linkedin.com/learning/benefits-of-github-community/discoverable-repositories?u=76281980&t=0)** - You've got a great open source project and you wanted to gain popularity and also look for others that could help you take it to the next level. Now, first, you should make sure that your repository is public. You determine repository visibility when you first create the project. However, you can change that afterwards and turn a repository from public to private in your settings. Just keep in mind that if you change a project visibility, it does have some consequences. Forks from a private repository, for example, will also be made into their own private projects. Another thing that makes the repository more discoverable is the project's name. A good repository name should have keywords that make it relevant for people searching for projects. So given a hint of what the project does without making it too long will make it stand out during searches. Names on [[GitHub]] are often written as lowercase characters with words separated by hyphens. You may also see some packages with uppercase letters or underscores, but you can use spaces or other special characters. As you can see when you create a project, you should also add a description which can also be a signal that will appear when people are searching for projects. They can only be about 350 characters long, but I would stick to something shorter because they can be truncated upon search. You'll also need a license that lets people know how you expect them to use and reuse the code. You may want to consider other files like the code of conduct to make your project more friendly. Once your main project is created, you could add a few more things to make it more searchable. Under the About section, you can add a link which might point to an official project site or demo. You can also add topic tax that will help people during the project search.
>
> **[1:32](https://www.linkedin.com/learning/benefits-of-github-community/discoverable-repositories?u=76281980&t=92)** There's also a way to add an image to your repository so that when people share a project, it automatically gets picked up and added to the post. Now a good README file is your project's first impression to the world. So make sure you add things like a summary of what your project does, installation instructions and samples of how to use the tool. GitHub makes it easy to add images and even video through drag and drop. That helps you add a demo or an illustration directly on your README file. You can also create a wiki which is the easiest way to create documentation, or create a GitHub page, which can be as simple as a docs folder with a number of files. You should consider using releases for your projects so that you can have an official version of your tools as well as a change log document which shows what's changed between versions. It's a lot of work, but the more of these features you add, the easier it will be to find an audience and get other developers to help you with your repo.
