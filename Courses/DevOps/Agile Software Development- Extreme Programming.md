> **[0:01](https://www.linkedin.com/learning/agile-software-development-extreme-programming/extreme-programming-xp-themes?u=76281980&t=1)** - [Narrator] The primary theme of XP is that if something hurts, do it all the time. And another way of saying that is if something's good, if it adds value, do it all the time. Woody Zuill says it like this, he says, "Turn up the good." And what he means is that if we notice something that's benefiting the team, some practice that we're getting good from, amplify that good and don't just turn it up a little bit, turn it up all the way to 11. For example, testing is good. It helps us find bugs before we deliver the product to our customers. But testing at the end of a release is painful because while we though we were done developing the product, it turns out that we've found lots of defects that we have to fix. It wreaks havoc with our release schedule. We actually don't know when we will be done developing the product. So instead of waiting until the end of the project, we test all the time, and we call that test driven development. The same thing with [Code Review](../../Skills/Software%20Development/Code%20Review.md). Reviewing code is good. We find and fix bugs before we deliver them to our customers. But code review is hard. We review the code infrequently, maybe every couple of weeks, maybe once a month. It's a long and boring meeting. And when we do find bugs, it takes a long time to remember what I was thinking way back when I wrote the code. So we resolve this pain and we get the benefits of doing code review by doing it all the time, and we call it pair programming. Another example is integrating our code. But it's painful. If we only do it every couple of weeks,

It takes a lot of time and hard work to get the code back into a state where it will compile, let alone run and pass all the tests. > **[0:01](https://www.linkedin.com/learning/agile-software-development-extreme-programming/user-stories?u=76281980&t=1)** - [Instructor] Most stories follow a pattern or a template. At a minimum, they have a title, a beginning, a middle, and an end. In one template, a story starts with the words once upon a time. It ends with the words and they all lived happily ever after. Imagine telling a story to a small group of friends. How would you tell the story? What would they be doing as you tell them the story? When I watch people tell stories to each other, I notice the listeners making eye contact, nodding, agreeing with the storyteller. They ask questions, and they get clarifications, like is this the version of "The Three Little Pigs" where the wolf is a villain, or is this the one where he just has a bad cold and the pigs misunderstand him? [Storytelling](../../Skills/Web%20Development/Storytelling.md) is something that humans have been doing for as long as humans have existed. We sit in a circle together. One person starts telling a story. Maybe another person fills in a missing detail. A third person asks a question, and a fourth person answers that question. And maybe the next night a different storyteller starts and tells the same story. Eventually everyone in the group knows the same story, and they can tell it to each other the same way. This is what user stories are. They're a way to describe what your users want to be true about the software you're building for them in the future. When someone asks you to implement a feature, they sometimes tell you all the little details of the solution they think they want. When that happens, ask them this question: What problem do you want us to solve? Part of why user stories work is they leave options open. They avoid aiming you at a specific pinpoint solution.
>
> **[1:36](https://www.linkedin.com/learning/agile-software-development-extreme-programming/user-stories?u=76281980&t=96)** For complex problems, we won't know the exact solution until we've begun doing the work and learned more. And we learn more both by doing the work itself and by getting iterative feedback from our stakeholders. By discussing the story out loud, asking questions, and getting clarifications, everyone on the team knows who cares about what we're doing, what they want, and why they want it. This is much more effective than writing down a list of so-called requirements in a large written document and asking everybody to read it and implement exactly what it says. There's much more information retained about what the stakeholders really want us to build when we have conversations with them and with each other. User stories typically follow a template that's similar to the once upon a time kind of template that we hear in fairy tales or really in any good storytelling tradition. They start with a title to make the story easy to remember. Then they introduce the main character, the stakeholder that we're building the software for. They tell us what the stakeholder wants and why they want it. It makes it easier for us to talk about the whole system from the high level architecture all the way down to the individual lines of code. System metaphor guides the names we choose for the various pieces of the system and helps us choose the best module names, class names, variable names and method names. And it makes it easy for anyone on the team to understand the code that anyone else wrote. To create your system metaphor, use the core protocols to assemble your set of practices to get into a state of shared vision together as a team. And as a team, articulate your team vision and your product vision. Your product vision might be something like language learners become experts in days. That product vision is the overarching system metaphor for the product. As a team, identify the stakeholder types whose lives will benefit from the system you're building. Make sure the names of the stakeholder types match the metaphor for the language learning platform the two stakeholder types are learner and teacher. For a car buying marketplace, metaphor appropriate stakeholder types might be buyer and seller. As a team, further develop the metaphor by creating user stories for the various stakeholder types. What problems do they have? What challenges do they face? How will the product you're building help them have better lives? More complex code has more defects,

Simple design guides us toward the simplest solution, the one that has the least bugs and it's easiest to modify. Simpler code is easier to understand and less code has fewer bugs. It's easy to enhance and adapt, it's easy to add new user stories as your stakeholders needs change over time. Smaller code builds faster and runs faster. The feedback cycles from [Test-Driven Development](../../Skills/Software%20Development/Test-Driven%20Development.md) and continuous integration are faster. Less code results in a smaller product that's easier to deliver to your stakeholders. So the goal is the smallest amount of the simplest code that works.
