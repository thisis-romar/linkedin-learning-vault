#### [Abstractions: Pros and cons](https://www.linkedin.com/learning/playwright-essential-training-abstractions-fixtures-and-complex-scenarios/abstractions-pros-and-cons?u=76281980)

> [!transcript]- Transcript
>
> **[0:00](https://www.linkedin.com/learning/playwright-essential-training-abstractions-fixtures-and-complex-scenarios/abstractions-pros-and-cons?u=76281980&t=0)** - [Instructor] In this chapter, we'll focus on ways to make Playwright test framework more maintainable through abstractions. First, let's define abstractions. Simplified representations of more complex systems, ideas, or processes. They help us manage complexity by focusing on the essential details and ignoring the irrelevant ones. But there are things you should know. With every abstraction comes risk. Abstracting too little can mean that you have a lot of code duplication, whereas abstracting too much can lead to overly complex code and interactions. Let's start with a popular type of abstraction, page objects. Abstracting away DOM elements into a page object can be a great way to separate your test code and implementation code. What I mean by that is building out unique locators for the different elements on a webpage and calling those elements by the name that you created. For example, if I landed on a login page, I may have an email input, a password input, and a login button. Ideally, I could create three unique variable names, email, password, login, and have these locators live in a different file than the test I'm writing. This way, if the underlying locator changes, I actually don't have to change the test. I only need to update the page object itself.
>
> **[1:35](https://www.linkedin.com/learning/playwright-essential-training-abstractions-fixtures-and-complex-scenarios/abstractions-pros-and-cons?u=76281980&t=95)** This simplifies maintaining your test. My advice is, create a page object locator for any element you interact with. Don't feel like you have to create page objects for every single element. Just add them along as needed. This is also really helpful in the site you're testing against if they use components. It's okay to create a higher level abstraction for a component that is used across multiple pages. Another way to abstract logic from a test could be to create a group of actions and have these actions live within the page object. Looking at our previous example, what if we were to combine three locator actions, fill email, fill password, and click login, into a single function that could be called from a Playwright test we're writing? This would allow us to only have one line of code in order to log into a system with multiple steps. This could be a really positive thing if you're having to log in multiple times throughout your test. This also helps when adding these actions to a page object class. Thanks to TypeScript's auto complete functionality, you'll have quick access to it. This also will help you maintain your test as they grow. My advice is, creating functions for groups of actions can be really useful if you use them more than once in a test or across multiple tests. Just make sure your function is somewhat generic
>
> **[3:08](https://www.linkedin.com/learning/playwright-essential-training-abstractions-fixtures-and-complex-scenarios/abstractions-pros-and-cons?u=76281980&t=188)** and can be used for multiple scenarios. Another practice I've seen is taking the page object or abstractions a step further and adding assertions to the page object file within a group of actions.
