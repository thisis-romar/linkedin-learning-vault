So finding ways of speeding up workflow is essential to help users be more productive in their tasks. These actions can be in the interface, or in keyboard shortcuts like command S for save. They can also be invoked from a context menu, or in a menu bar. Some of these common actions are additive properties like add, copy, paste, create and new. These actions will likely be found in the menu bar for a software product, but can also be labeled actions on web application interfaces. For example, if you want to create something new in [Google Drive](../../Skills/Software%20Development/Google%20Drive.md), you could do that by pressing the button on the top left. Usually, the ability to do some of these additive properties are displayed as a main action, but your use case may vary. Another set of actions are removal actions, like undo, remove, close and delete. Undo is typically global, and then a user can take an action and use either a keyboard shortcut or find undo in the top menu bar. Undo is must have function in most applications. Users will lose confidence and trust in the application if they're not able to roll back the actions they've taken. Undo can be for a single action, but it's common to have the ability to undo multiple actions by continuing to undo every action taken. Remove, close and delete are actions that are contextual to the item being affected. For example, these actions might be seen

Autosave is a pattern that automatically saves the work the user's doing in the background, so the user doesn't have to do a hard save. This pattern is common for web applications, and is helpful to keep users focused on their task without worrying about a worst case scenario of losing their work. Auto Refresh refreshes areas of the screen automatically without any input from the user. Auto Refresh is useful for scenarios in which data becomes stale or needs to be updated regularly. It's also fairly common to show a button allowing the user to refresh manually, to avoid any confusion on what's changed on screen. Usually refresh and update buttons are in the context of the item or section that needs updating, or placed on the top of the page. When designing web applications and software, consider these common actions to enhance your experience.

#### [Guided tour](https://www.linkedin.com/learning/interaction-design-software-and-web-design-patterns/guided-tour?u=76281980)

> [!transcript]- Transcript
>
> **[0:00](https://www.linkedin.com/learning/interaction-design-software-and-web-design-patterns/guided-tour?u=76281980&t=0)** - [Instructor] Guided tours are a pattern that introduces users to new features or to the product. They can be helpful for features that might be hidden from the interface, or if the utility isn't well understood, or just to highlight the product. This pattern is characterized by a series of tooltips or modals, which highlight product features the user will find useful. Not every feature requires a guided tour. Some features are self-explanatory and don't need an introduction. If the tour is unhelpful it may be in the way of the user to have a tour pointing out basic product functionality. Guided tours typically have a series of popovers, tooltips, or modals that indicate the number of steps in the tour. Usually there's an option to skip forward and backward or to cancel out of the tour. I recommend using a back button on a tour, because sometimes users will instinctively click on the next button, missing the previous step, and will need a way to view the step again. Guided tours are best triggered by events for when and how they populate. For example, you can set up rules to only display the tour the first time the user logs into the product. Or if the user hasn't logged in for a while and you have multiple features turned on since their last login. You can design rules on which features take precedent to highlight, so they aren't shown multiple tours at the same time. You'll also want to set rules for when the user has dismissed the tour, so it's not displayed every time the user logs in.
