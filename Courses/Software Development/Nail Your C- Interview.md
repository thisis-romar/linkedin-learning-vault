Class libraries, like list, are what make up the .NET framework library. .NET makes it easier to get something up and running with the C# programming language. To bring in an analogy, you can think of C# as the airplane and .NET is the runway. C# is a programming language, while .NET is a developer platform. This means that knowledge of C# also implies some knowledge of .NET. However, the opposite isn't necessarily true, since you can use other languages to write .NET applications.

#### [Managed vs. unmanaged code](https://www.linkedin.com/learning/nail-your-c-sharp-interview/managed-vs-unmanaged-code?u=76281980)

> [!transcript]- Transcript
>
> **[0:00](https://www.linkedin.com/learning/nail-your-c-sharp-interview/managed-vs-unmanaged-code?u=76281980&t=0)** - The difference between managed and unmanaged code has to do with how the code is built and executed. This has important security implications which is why it's often asked in interviews. Let's start with managed code. Managed code is code where the execution is managed by the runtime. In our case, that's the CLR, the common language run time. The CLR is in charge of taking managed code, compiling it into machine code, and then executing it. As we discussed before, the common language run time also provides several important services such as garbage collection, security boundaries, and type safety. But it happens to be for this managed code. It does all of this without the interference of the programmer. Human error is often the cause of most bugs. In an unmanaged world, the developer has to do a lot more work. They're in charge of [Memory Management](../../Skills/Web%20Development/Memory%20Management.md), security considerations, building the binary, the OS loads into memory and more. This is the way you would write and run a C or C ++ program. Often unmanaged code is aimed at a specific processor architecture and depends on the machine. This means when you compile unmanaged code,
>
> **[1:34](https://www.linkedin.com/learning/nail-your-c-sharp-interview/managed-vs-unmanaged-code?u=76281980&t=94)** it's native to that platform. If you want to use the same code on a different architecture, you'll need to recompile that same code again. So why would you ever use unmanaged code? Well, unmanaged code allows you to get low level access to the CPU architecture. You have more control over how your memory is allocated, which is necessary for some applications. It also allows you to bypass some parameters and restrictions, but you would need to do so with caution. Now, the common language run time allows you to pass between managed and unmanaged code. There are some base libraries that do this where you wrap an unmanaged library and call into it. However, it is important to note that once you do this, the actual execution for this portion of the program is in the hands of the unmanaged code. While you can use unmanaged constructs in C#, it's typically not recommended for the everyday programmer due to the security consequences. It's often too much effort to maintain and secure unless you're working with low level processing. And in that case, you should consider whether C# is the correct language for the problem you are solving.

This is known as data hiding. In C#, we can achieve this by declaring all the fields in a class as private and writing public methods in the class to get and set the values of the variables. With this, other classes can still access the hidden data, but they can only do so through a public method of the given class. Let's take a look at an example. Here, we have a HogwartsStudent with name and HogwartsHouse data. HogwartsHouse is just an enum with four different Hogwart houses that a student could be assigned to. This student class generates the house piece of data using the int value of the name in the constructor. The class also provides a few public getter methods
>
> **[1:37](https://www.linkedin.com/learning/nail-your-c-sharp-interview/add-encapsulation-to-your-programs-to-impress-interviewers?u=76281980&t=97)** to access the data. Since the data pieces are private, the only way to access the data is through the public methods. You might also notice there are no setters in this class. We do this so that HogwartsStudent instances become immutable once created. No changing houses or names. This class encapsulates the private data pieces that are inside of it. Now let's use it. We'll create a student and print out the name and the house. To access the different data pieces within the student object, we have to use the public methods or the public getters provided in the HogwartsStudent class. Here, we use getName and getHouse to get the data, because we cannot directly access or modify the data. If we try to access it directly, we'll get some errors. This is because each piece of data has the private access modifier. The student class encapsulates the data inside of it. We hide the implementation of the name and house data through the public methods. Now, why do this? We use encapsulation so the user has no idea of the inner implementation of the class and the data it contains. The user doesn't need to know
>
> **[3:10](https://www.linkedin.com/learning/nail-your-c-sharp-interview/add-encapsulation-to-your-programs-to-impress-interviewers?u=76281980&t=190)** how the student gets assigned to a house, and it doesn't have access to that data directly. The user only knows how to retrieve data using the getter method. This type of implementation can help us keep classes separated. It's important to remember that encapsulation is different from abstraction. Abstraction provides generalization, while encapsulation hides unwanted implementation details from the actual users of an object. We write encapsulated code to improve the reusability of our program. We want to make it easy to change when new requirements come along. What happens if we need to add a new HogwartsHouse? Only one class needs to be changed in this case, rather than multiple. Encapsulation also makes [Unit Testing](../../Skills/Software%20Development/Unit%20Testing.md) much easier, because each piece is less coupled.
