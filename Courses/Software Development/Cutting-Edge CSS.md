>
> **[0:27](https://www.linkedin.com/learning/cutting-edge-css/what-are-logical-properties-14445691?u=76281980&t=27)** Now, this box model is built on an assumption that text is always displayed on horizontal lines flowing left to right.

> **[0:43](https://www.linkedin.com/learning/cutting-edge-css/what-are-logical-properties-14445691?u=76281980&t=43)** And when the text direction changes, the traditional box model and its properties become illogical.
>
> **[0:49](https://www.linkedin.com/learning/cutting-edge-css/what-are-logical-properties-14445691?u=76281980&t=49)** With vertical text, this whole width and height and top, right, bottom, left way of thinking about the box gets confusing because the end of the line is now at the bottom, not the right, and the right side of the box is to the right of the first line of text, which is not the way we normally think about things.
>
> **[1:06](https://www.linkedin.com/learning/cutting-edge-css/what-are-logical-properties-14445691?u=76281980&t=66)** Here's the thing.
>
> **[1:07](https://www.linkedin.com/learning/cutting-edge-css/what-are-logical-properties-14445691?u=76281980&t=67)** Working with text, we don't set the width of a text.
>
> **[1:11](https://www.linkedin.com/learning/cutting-edge-css/what-are-logical-properties-14445691?u=76281980&t=71)** We define the length of the line.
>
> **[1:13](https://www.linkedin.com/learning/cutting-edge-css/what-are-logical-properties-14445691?u=76281980&t=73)** And we don't set the height of the text box.
>
> **[1:15](https://www.linkedin.com/learning/cutting-edge-css/what-are-logical-properties-14445691?u=76281980&t=75)** We set the length of the axis perpendicular to the line direction.

> **[1:19](https://www.linkedin.com/learning/cutting-edge-css/what-are-logical-properties-14445691?u=76281980&t=79)** This is why we now have logical properties.
>
> **[1:23](https://www.linkedin.com/learning/cutting-edge-css/what-are-logical-properties-14445691?u=76281980&t=83)** With logical properties, we control the layout of the box based on the logical flow of its contents rather than their physical direction.
>
> **[1:30](https://www.linkedin.com/learning/cutting-edge-css/what-are-logical-properties-14445691?u=76281980&t=90)** So instead of thinking about the width and height of a box, we think about the inline and block size of the box, inline being the inline dimension or axis the text flows along, and block being the dimension or axis perpendicular to inline.
>
> **[1:46](https://www.linkedin.com/learning/cutting-edge-css/what-are-logical-properties-14445691?u=76281980&t=106)** With logical properties, we set the inline size, the length along the inline axis, with the flow of the text, and the block size, the length perpendicular to that inline axis, so effectively the number of lines allowed in the box.
>
> **[2:00](https://www.linkedin.com/learning/cutting-edge-css/what-are-logical-properties-14445691?u=76281980&t=120)** And for the sides, the padding and border and margin, the logical properties become inline-start and inline-end, block-start and block-end.
>
> **[2:10](https://www.linkedin.com/learning/cutting-edge-css/what-are-logical-properties-14445691?u=76281980&t=130)** That way, when the text direction changes or writing mode changes from left to right to right to left or from horizontal to vertical, the logical properties follow suit, and the box size, padding, border, and margin all behave, well, logically.

>
> **[0:23](https://www.linkedin.com/learning/cutting-edge-css/why-margins-collapse?u=76281980&t=23)** So I'll set margin blocks, start on the second one to one REM and nothing happens.

>
> **[0:51](https://www.linkedin.com/learning/cutting-edge-css/why-margins-collapse?u=76281980&t=51)** And even though it may look like a bug in CSS, it actually makes sense.

> **[0:55](https://www.linkedin.com/learning/cutting-edge-css/why-margins-collapse?u=76281980&t=55)** The key is to think about the origin purpose of CSS, styling text in documents.
>
> **[1:01](https://www.linkedin.com/learning/cutting-edge-css/why-margins-collapse?u=76281980&t=61)** Let's expand our example.
>
> **[1:03](https://www.linkedin.com/learning/cutting-edge-css/why-margins-collapse?u=76281980&t=63)** We have a heading, two paragraphs and an image.
>
> **[1:06](https://www.linkedin.com/learning/cutting-edge-css/why-margins-collapse?u=76281980&t=66)** We want consistent rhythm between these elements.
>
> **[1:09](https://www.linkedin.com/learning/cutting-edge-css/why-margins-collapse?u=76281980&t=69)** And we don't know how many paragraphs or headings we will have in the document.
>
> **[1:14](https://www.linkedin.com/learning/cutting-edge-css/why-margins-collapse?u=76281980&t=74)** Now, the heading and paragraphs have default block start and block end margins.
>
> **[1:18](https://www.linkedin.com/learning/cutting-edge-css/why-margins-collapse?u=76281980&t=78)** The image has no margins.
>
> **[1:20](https://www.linkedin.com/learning/cutting-edge-css/why-margins-collapse?u=76281980&t=80)** Now watch what happens when we put them together.
>
> **[1:23](https://www.linkedin.com/learning/cutting-edge-css/why-margins-collapse?u=76281980&t=83)** If margins did not collapse, here's what we get.
>
> **[1:26](https://www.linkedin.com/learning/cutting-edge-css/why-margins-collapse?u=76281980&t=86)** As you can see, there is inconsistency in the spacing between the items in particular between the paragraphs and the image.
>
> **[1:33](https://www.linkedin.com/learning/cutting-edge-css/why-margins-collapse?u=76281980&t=93)** That's because the paragraph margins are set up so the paragraph has enough block level white space above and below, regardless of what the next or previous element is.

> **[1:43](https://www.linkedin.com/learning/cutting-edge-css/why-margins-collapse?u=76281980&t=103)** And when margins don't collapse, that space is doubled up between each paragraph.
>
> **[1:48](https://www.linkedin.com/learning/cutting-edge-css/why-margins-collapse?u=76281980&t=108)** By collapsing the margins, meaning the browser picks whichever margin is greatest and overlaps it on top of the other one, we get consistent spacing between all elements.
>
> **[1:59](https://www.linkedin.com/learning/cutting-edge-css/why-margins-collapse?u=76281980&t=119)** That way, if we have a layout like this, the rhythm of the spacing matches.
>
> **[2:03](https://www.linkedin.com/learning/cutting-edge-css/why-margins-collapse?u=76281980&t=123)** And if we move the image up between the paragraphs, the rhythm remains because now the previously collapsed margins in the paragraphs un-collapse and work as actual margins.
>
> **[2:17](https://www.linkedin.com/learning/cutting-edge-css/why-margins-collapse?u=76281980&t=137)** In short, margins collapse to make elements more reusable.
>
> **[2:21](https://www.linkedin.com/learning/cutting-edge-css/why-margins-collapse?u=76281980&t=141)** Otherwise we'd have to specify rules, like "add a one REM margin only to the block start of the first paragraph, except when the previous element doesn't have a block in the margin, in which case add a one REM margin."
>
> **[2:34](https://www.linkedin.com/learning/cutting-edge-css/why-margins-collapse?u=76281980&t=154)** Which would be really convoluted and error prone.
