The difference between the two project types are primarily about how the code is structured and the specific features that are available. Controller-based APIs are structured, as the name suggests, around controller classes. You'll find that these are structured and work a lot like Model-View-Controller apps without the View. #### [Understanding gRPC APIs](https://www.linkedin.com/learning/learning-asp-dot-net-core-mvc-razor-pages-web-apis-other-foundations/understanding-grpc-apis?u=76281980)

> [!transcript]- Transcript
>
> **[0:00](https://www.linkedin.com/learning/learning-asp-dot-net-core-mvc-razor-pages-web-apis-other-foundations/understanding-grpc-apis?u=76281980&t=0)** - [Instructor] Standard web APIs like we saw in the previous chapter aren't the best solution for every scenario. Sometimes you need a little more performance. gRPC is a remote procedure call framework that is language agnostic and can give you better performance than a simple web API. By default, gRPC services utilize [Protocol Buffers](../../Glossary/Standard/Protocol%20Buffers.md), which is a language-neutral platform developed by [Google](../../Glossary/Service/Google.md) for serializing structured data. gRPC services with protocol buffers are developed contract first. This means that services are defined independent of any programming language in special proto files. They define the calls you make to the service, as well as the types of parameters they accept and the data they return. gRPC is language agnostic. Tooling exists in lots of different languages to read the proto files that define a service and generate strongly typed server code to implement the service and client code to call it. gRPC services generally perform better than web APIs because of the reduced network usage that results from the [protobuf](../../Glossary/Standard/Protocol%20Buffers.md) binary serialization. For all these reasons, gRPC services can be a good alternative if you need the flexibility to work with different languages, need to deliver services that are more efficient and performant, or need more advanced [Real-Time](../../Skills/Database%20Management/Real-Time.md) communication than you would get with a typical [rest](../../Skills/Web%20Development/Representational%20State%20Transfer%20(REST).md) service. > **[0:00](https://www.linkedin.com/learning/learning-asp-dot-net-core-mvc-razor-pages-web-apis-other-foundations/understanding-signalr-applications?u=76281980&t=0)** - [Instructor] Server-side web apps and services are great at sitting around waiting on request and then sending responses to those requests back to the client. However, that means the client won't know about new information unless they request and receive it. SignalR adds [Real-Time](../../Skills/Database%20Management/Real-Time.md) web functionality to apps and gives servers the ability to push content to clients at any time. This can be great for things like real-time chat apps, corporate [Dashboards](../../Skills/Data%20Science/Dashboards.md) that always need the latest data, and any app requiring time-sensitive alerts. SignalR accomplishes this by supporting two-way communication between the server and clients. The server acts as a hub for this communication and can push messages to all clients or individual clients as needed. [ASP.NET](https://ASP.NET) Core even provides a base class named hub you can use to implement your SignalR servers. SignalR supports multiple transports for the communication between the server and clients. [WebSockets](../../Skills/Web%20Development/WebSocket.md) are the preferred transport, and will automatically be used if supported by both the client and server. This shouldn't be a problem with any modern browser, since they all support WebSockets these days. However, SignalR will gracefully fall back to using [Server-Sent Events](../../Glossary/Standard/Server-Sent%20Events.md), or long polling if a particular client doesn't support WebSockets. The last thing I'll mention before I get into a demo is that SignalR is really a feature you can add to any [ASP.NET](https://ASP.NET) Core project.
>
> **[1:34](https://www.linkedin.com/learning/learning-asp-dot-net-core-mvc-razor-pages-web-apis-other-foundations/understanding-signalr-applications?u=76281980&t=94)** There's relatively little server code required to get a basic SignalR hub up and running, so I would really think of it less like another project type and more of a feature you can add to any project when you need real-time functionality.
