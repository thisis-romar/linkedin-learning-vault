#### [Understanding relational databases](https://www.linkedin.com/learning/databases-for-node-js-developers-25329301/understanding-relational-databases?u=76281980)

> [!transcript]- Transcript
>
> **[0:00](https://www.linkedin.com/learning/databases-for-node-js-developers-25329301/understanding-relational-databases?u=76281980&t=0)** - [Instructor] When I first started using [Databases](../../Skills/Software%20Development/Databases.md) professionally, [Relational Databases](../../Skills/Software%20Development/Relational%20Databases.md) were the standard. Back then, [MySQL](../../Skills/Software%20Development/MySQL.md) and [Postgres](../../Skills/Software%20Development/PostgreSQL.md) dominated the space and they're still widely used today. A relational database is great when data has clear relationships. Imagine building a hiking app that answers questions like which villages are along a trail? Where can a eat in a village? Which trail leads to a specific village? These questions involve relations between data, and that's exactly what relational databases are designed for. Relational databases follow a strict schema, which defines tables, structured storage for data, columns, which define what each record holds, and relationships, how records are linked. Let's look at the few relationship types. We have one-to-one. For instance, a student has one place of birth. We have one-to-many relationships, which would be a college has many students, and we have many-to-many relationships, a subject has many students and the student has many subjects. To design relational databases, we use entity relationship diagrams, ERDs. These visually present how data connects. For example, in our hiking app, a trail can pass through multiple villages, a village can have multiple trails, and a trail can have many points of interest.
>
