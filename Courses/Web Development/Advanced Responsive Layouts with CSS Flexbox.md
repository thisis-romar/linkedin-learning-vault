#### [Caveats when using Flexbox today](https://www.linkedin.com/learning/advanced-responsive-layouts-with-css-flexbox/caveats-when-using-flexbox-today?u=76281980)

> [!transcript]- Transcript
>
> **[0:00](https://www.linkedin.com/learning/advanced-responsive-layouts-with-css-flexbox/caveats-when-using-flexbox-today?u=76281980&t=0)** - Before we get into the nitty gritty, I'd be remiss if I didn't bring up a couple of really important caveats when we're working with Flexbox. As you can see here, on [http://canuise.com](http://canuise.com), Flexbox is currently supported in all modern current browsers. But if you step back just one browser version, you'll see there's a lot less support, or no support at all. This is because Flexbox is about as bleeding edge as you can get with CSS. In fact, it's not fully ratified yet, and there's some components that are not working in any browsers because no one can agree on how they work. And even in modern browsers, there are some browsers, specifically Safari, that require prefixing to work. That means, if you're gonna use Flexbox in a current live project on the web, you need to make your site fully backwards compatible to browsers that either don't support Flexbox, or don't support the current markup for Flexbox. In fact, the current recommendation is to use Flexbox as a progressive enhancement for modern browsers, and build your site to work without Flexbox for everyone else. Now when you're building a site using Flexbox, I have a couple tools I want to draw your attention to. I'm not going to use those tools in this course, because this course is only about Flexbox and Flexbox markup. But as you work in your own projects, you should be using these tools to ensure backwards compatability. The first tool is called the Autoprefixer. It's a script that automates the process of prefixing your CSS. It doesn't just work on Flexbox. It addresses anything that needs to be prefixed, and it ensures that you don't have to remember how to write the prefixes for all the different older browsers. Autoprefixer can be run on its own,
>
> **[1:34](https://www.linkedin.com/learning/advanced-responsive-layouts-with-css-flexbox/caveats-when-using-flexbox-today?u=76281980&t=94)** or you can incorporate it into your favorite task runner, including Grunt and Gulp, if you want to. You also want to use Modernizr. Now Modernizr is a clever [JavaScript](../../Skills/Software%20Development/JavaScript.md) that sits at the top of your page. And all it does is track whether or not certain features are supported by the current browser that the page is opened in. If a feature is not supported, like, for example, Flexbox Modernizr will add in a class at the top of your body tag, so that you can create custom styles for just that class. Using Modernizr, you can then create whole stacks of CSS that will only be called in either if Flexbox is supported, or if Flexbox is not supported by a certain browser. And you can build functional, modularized CSS, that targets individual browser types, and provide a good [user experience](../../Skills/Web%20Development/User%20Experience%20(UX).md) for all visitors. So as we're diving into this, make sure you'll always remember that what we're looking at right now is cutting edge CSS that may not work in all browsers. That does not mean you shouldn't use it. I urge you to start using Flexbox today. However, make sure you have full backwards compatibility, at least one step back, so that people who come to visit your site can actually get a good user experience out of it.
